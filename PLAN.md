## 1. 프로젝트 개요

### 1.1 한 줄 소개

**"피부타입별 화장품 추천 API를 최종 산출물로, 실시간 + 배치 데이터 파이프라인을 GCP 위에 코드로(IaC) 구축하고 CI/CD와 보안 검사가 자동화된 프로덕션 수준의 데이터 플랫폼"**

### 1.2 왜 이 프로젝트인가?

주니어 포트폴리오의 흔한 실패 패턴은 "튜토리얼 따라하기"로 보이는 것이다.
기업이 실제로 확인하고 싶은 것은 다음 3가지이며, 이 프로젝트는 그것을 직접 보여준다.

| 기업이 보는 것 | 이 프로젝트에서 증명하는 방법 |
|---|---|
| 데이터가 흐르는 **전체 구조**를 이해하는가 | 수집 → 저장 → 처리 → 적재 → 시각화 End-to-End 직접 구축 |
| **운영(Ops)** 을 고려하는가 | IaC, CI/CD, 모니터링, 알림, 장애 대응 문서화 |
| **비용과 보안**을 의식하는가 | 예산 알림, 최소 권한 IAM, 시크릿 관리, 취약점 스캔 |

### 1.3 채용 JD 공통 요구사항 → 기술 매핑

국내 중견·대기업(네이버, 카카오, 쿠팡, 토스, 배민, LG CNS, 삼성SDS 등) 데이터 엔지니어/DevOps JD에서 반복 등장하는 키워드를 기준으로 기술을 선정했다.

| JD 키워드 | 채택 기술 | 이 프로젝트에서의 사용처 |
|---|---|---|
| Python, SQL | Python 3.12, BigQuery SQL | 모든 파이프라인 코드, 데이터 변환 |
| 워크플로우 오케스트레이션 (Airflow) | **Cloud Composer 대신 GCE/로컬 Airflow** (비용 절감) | 배치 파이프라인 스케줄링 |
| 스트리밍 (Kafka, Kinesis) | **Pub/Sub** (GCP 관리형, Kafka와 동일 개념) | 실시간 이벤트 수집 |
| 분산 처리 (Spark) | **Dataproc Serverless** 또는 로컬 PySpark | 대용량 배치 변환 |
| 데이터 웨어하우스 (BigQuery, Redshift, Snowflake) | **BigQuery** | 최종 분석용 데이터 저장소 |
| 데이터 모델링 (dbt) | **dbt** | DW 내 변환 계층 (staging → mart) |
| 컨테이너 (Docker, Kubernetes) | **Docker + GKE Autopilot** (또는 Cloud Run) | 수집기/API 배포 |
| IaC (Terraform) | **Terraform** | 모든 GCP 리소스를 코드로 관리 |
| CI/CD (GitHub Actions, Jenkins) | **GitHub Actions** | 테스트 → 빌드 → 스캔 → 배포 자동화 |
| 모니터링 (Prometheus, Grafana) | **Cloud Monitoring + Grafana** | 파이프라인 상태/비용 대시보드 |
| DevSecOps (SAST, 이미지 스캔, 시크릿 관리) | **Trivy, Bandit, gitleaks, Secret Manager** | CI 파이프라인 내 보안 게이트 |

> ⚠️ **원칙**: JD에 나온다고 무작정 다 쓰지 않는다. "왜 이 기술을 선택했는지" 설명 가능한 것만 쓴다. (면접 단골 질문)

## 2. 프로젝트 주제

### 주제: **"피부타입별 화장품 추천 데이터 플랫폼"**

화장품 제품·성분 데이터(배치)와 사용자 행동 이벤트(스트리밍)를 결합해, **피부타입별 화장품 추천 API**를 최종 산출물로 제공하는 데이터 플랫폼.

#### 데이터 소스 전략 (이 주제의 핵심 설계 결정)

| 데이터 | 소스 | 수집 방식 | 비고 |
|---|---|---|---|
| 제품·성분 마스터 | 공공데이터포털 (식약처 화장품 제품/성분/기능성 데이터) | **배치** (일 1회 API 수집) | 무료 공식 API. 신뢰 가능한 원천 |
| 사용자 행동 이벤트 (조회/찜/구매) | **가상 이벤트 생성기** (피부타입별 사용자 프로필 시뮬레이션) | **스트리밍** (Pub/Sub) | 실제 기업의 클릭스트림 파이프라인과 동일 구조 |
| 리뷰·평점 (선택) | 크롤링은 약관 위반 소지로 **배제** | — | 대신 이벤트 생성기에 평점 이벤트 포함 |

> ⚠️ **크롤링을 쓰지 않는 이유(면접 답변 포인트)**: 화해/올리브영 리뷰 크롤링은 서비스 약관 위반 소지가 있다.
> 대신 피부타입별 선호 패턴을 반영한 **합성(synthetic) 이벤트 생성기**를 설계했다 — 이벤트 스키마 설계,
> 트래픽 패턴 시뮬레이션, 스트리밍 처리라는 실무 역량을 합법적으로 증명하는 선택이다.

- **배치 파이프라인**: 매일 식약처 API에서 제품/성분 데이터 수집 → GCS(데이터 레이크) → BigQuery 적재 → dbt 변환
- **스트리밍 파이프라인**: 이벤트 생성기(Cloud Run) → Pub/Sub → BigQuery 스트리밍 적재 → 실시간 인기 지표
- **추천 로직**: ML이 아닌 **dbt/SQL 기반** — 성분 유사도 + 피부타입별 인기 집계 (본질은 데이터 엔지니어링)
- **서빙**: "피부타입 X에 맞는 상품 추천" REST API (FastAPI, Cloud Run 배포)

> 주제는 수단이다. 핵심은 **파이프라인·인프라·운영 역량**이며, 주제가 바뀌어도 구조는 재사용 가능하게 만든다.

## 7. 성공 기준 (완성의 정의)

1. `terraform apply` 한 번으로 인프라 전체가 재현된다.
2. 배치 파이프라인이 **7일 이상 무인 운영**되고, 실패 시 알림이 온다.
3. PR을 올리면 테스트 + 보안 스캔이 자동으로 돌고, 통과해야 병합된다.
4. README만 읽고 제3자가 아키텍처와 기술 선택 이유를 이해할 수 있다.
5. 모든 트러블슈팅이 docs/에 기록되어 면접 답변 소재가 된다.
