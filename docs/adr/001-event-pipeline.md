# ADR 001 — 사용자 행동 이벤트 파이프라인 설계

> 상태: 채택 (2026-07-08) · Phase 2

## 상황

피부타입별 추천을 만들려면 "누가 무엇을 봤/샀/평가했는지" 행동 데이터가 필요하다.
크롤링은 배제(약관 리스크, PLAN 2장)했으므로 **가상 이벤트 생성기**가 원천이 된다.
이벤트를 Pub/Sub에서 BigQuery까지 나르는 경로를 정해야 한다.

## 결정 1: 이벤트 스키마 (v1)

```json
{
  "event_id":  "uuid — 중복 제거 키 (Pub/Sub는 at-least-once라 중복 배달이 정상)",
  "event_type": "view | like | purchase | rating",
  "event_ts":  "ISO8601 UTC — 이벤트 발생 시각 (발행 시각과 구분!)",
  "user_id":   "u_00042",
  "skin_type": "dry | oily | combination | sensitive | normal",
  "product_id": "p_0137",
  "rating":    "1~5, rating 이벤트에만 존재",
  "session_id": "s_... — 같은 방문 안의 행동 묶음"
}
```

- **event_id 필수**: Pub/Sub는 "최소 1회 배달"을 보장할 뿐 "정확히 1회"가 아니다.
  중복 제거는 하류(dbt)에서 event_id로 한다 — 스트리밍의 기본 상식.
- **skin_type을 이벤트에 포함**: 정규화하면 users 테이블이 따로 필요하지만,
  클릭스트림에서는 조회 시점 속성을 이벤트에 박아 보내는(denormalize) 것이 일반적.

## 결정 2: 수집 경로 — Pub/Sub **BigQuery 구독** (Dataflow 배제)

| 선택지 | 비용 | 복잡도 | 판단 |
|---|---|---|---|
| Dataflow (Beam) | 상시 워커 = 월 수십만 원 | 높음 | 지금 규모(초당 수 건)에 과잉 — JD에 자주 나오지만 "왜 안 썼나"를 답할 수 있는 게 더 가치 있음 |
| 자체 컨슈머 (Cloud Run) | 낮음 | 중간 | 직접 운영할 코드가 늘어남 |
| **BigQuery 구독** | 사실상 0 (10GB/월 무료) | 최소 | ✅ 관리형이 배달·재시도·적재를 다 해줌 |

- 메시지는 events 테이블의 `data` 컬럼(STRING)에 원문 그대로 적재 → **raw 원형 보존 원칙과 일치**.
  JSON 파싱·타입 정리는 dbt staging의 책임 (배치와 같은 구조).
- 변환이 필요한 규모/요구가 생기면 그때 Dataflow를 넣는다 — "필요해질 때까지 안 만든다".

## 결정 3: 가상 카탈로그

제품↔성분 매핑 데이터가 공공에 없으므로, **실제 성분(stg_ingredient)에서 추출한 이름으로
합성 제품 카탈로그(300개)를 생성**한다. 시드 고정(재현 가능), 피부타입별 타깃/주의 성분 포함.
→ 추천 mart가 "실제 규제 데이터 × 합성 카탈로그 × 합성 행동"을 조인하는 구조가 된다.

## 트레이드오프

- BigQuery 구독은 메시지를 변환 없이 넣는다 → 스트리밍 중 변환(윈도 집계 등) 경험은 이 프로젝트 범위 밖
- 합성 데이터의 분포는 설계자가 정한 것 — 추천 "정확도"는 평가 대상이 아님을 README에 명시
