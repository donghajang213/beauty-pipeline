# 그린필드 플레이북 — 아무것도 없는 회사에서 데이터 플랫폼을 세우는 순서

> **누구를 위한 문서**: "입사했더니 데이터 파이프라인이 없다"는 상황을 만난 (미래의) 나.
> 이 저장소가 그 축소판 실습이었다 — 각 단계마다 "우리 프로젝트에선 이렇게 했다"를 증거로 단다.

## 0단계 — 코드보다 먼저: 질문 4개

구축 순서는 기술이 아니라 **질문의 답**이 정한다.

| 질문 | 우리 프로젝트의 답 | 현업에서 |
|---|---|---|
| 데이터의 **소비자**가 누구고 무엇을 원하나 | 추천 API와 대시보드 | 경영 리포트? ML팀? 서비스 기능? — 소비자 없이 만든 파이프라인은 유령이 된다 |
| **원천**은 무엇이고 어떤 성질인가 | 공공 API(마스터성, 일 갱신) + 행동 이벤트(실시간) | 사내 DB, SaaS(광고/CRM), 로그 — 각각 배치/CDC/스트리밍이 갈린다 |
| **SLA**: 얼마나 신선해야 하나 | 일 배치 + 이벤트는 준실시간 | "실시간이요"는 대부분 재질문하면 "하루 한 번"이다 — 비용이 10배 갈리는 질문 |
| **예산/팀 규모**는 | 1인, 크레딧 43만 원 | 관리형 vs 자체 운영의 갈림길. 작을수록 관리형 |

## 구축 순서 (우리가 밟은 순서 그대로)

### 1. 안전장치와 땅고르기 — terraform/, 예산 알림
코드 한 줄 전에 **예산 알림**(우리는 gross 기준으로 교정)과 **IaC 부트스트랩**(state 버킷 → 나머지 전부 코드로).
현업 추가사항: 조직/폴더 구조, 환경 분리(dev/prod 프로젝트), 팀 IAM 그룹. 원칙은 동일 — 콘솔 클릭으로 만든 리소스는 나중에 아무도 설명 못 한다.
→ 예시: [terraform/](../terraform/), [docs/setup/00-bootstrap.md](setup/00-bootstrap.md)

### 2. 수집 계층 — ingestion/
원천별 커넥터에 **재시도(백오프)·타임아웃·멱등성**부터 심는다. 신규 원천은 probe(탐침)로 실호출 검증 후 레지스트리에 등록.
현업 추가사항: 원천이 사내 DB면 CDC(Datastream 등)나 스냅샷 덤프, SaaS면 관리형 커넥터(Fivetran류) 구매 검토 — "만들 것인가 살 것인가"가 첫 결정.
→ 예시: [ingestion/datago.py](../ingestion/datago.py) (커넥터), [ingestion/sources.py](../ingestion/sources.py) (레지스트리)

### 3. 레이크 + DW — 저장은 두 겹
원형은 레이크(dt= 파티션, 수명 정책), 분석은 DW(파티션 테이블, load job). **원형 보존이 재처리의 보험**이다.
현업 추가사항: 데이터셋 명명 규칙과 소유권(도메인별), PII 컬럼 정책(마스킹/토큰화)을 초기에 정해야 나중에 안 터진다.
→ 예시: [terraform/storage.tf](../terraform/storage.tf), [ingestion/batch/load_to_bq.py](../ingestion/batch/load_to_bq.py)

### 4. 변환 계층 — dbt/
raw는 손대지 않고 staging(이름·타입·중복)→marts(비즈니스 로직)로. **테스트는 프로파일링으로 확인한 사실에만.**
현업 추가사항: 지표 정의의 주인(마트)을 정해 부서마다 다른 숫자가 나오는 사태를 막는다. dbt docs로 계보 공유.
→ 예시: [dbt/models/](../dbt/models/) — 실결함 176건을 테스트가 잡은 기록 포함

### 5. 오케스트레이션 — airflow/
손으로 두 번 이상 실행한 순서는 DAG로. 재시도는 코드(초 단위)와 오케스트레이터(분 단위) 2층으로.
현업 추가사항: Composer/MWAA 같은 관리형 채택(팀이면 그만한 가치), 실패 알림을 Slack/PagerDuty에 연결, backfill 정책 합의.
→ 예시: [airflow/dags/beauty_daily.py](../airflow/dags/beauty_daily.py) — 실장애 5건 자동 극복 실측

### 6. 스트리밍 — 필요해질 때
"진짜 실시간이 필요한가"를 먼저 의심하고, 필요하면 관리형 최소 구성부터(Pub/Sub → BigQuery 구독). event_id 중복 제거는 스키마 설계 단계에서.
현업 추가사항: 이벤트 스키마 거버넌스(스키마 레지스트리, 버전 정책)가 팀 규모에서는 필수.
→ 예시: [docs/adr/001-event-pipeline.md](adr/001-event-pipeline.md) — Dataflow를 "안 쓴" 근거까지

### 7. 서빙 — api/, 대시보드
계산은 마트가 미리, 서빙은 읽기만(사전 계산 패턴). BI는 전용 마트 뷰만 보게.
→ 예시: [api/main.py](../api/main.py), [docs/dashboard.md](dashboard.md)

### 8. 보안·운영은 "마지막"이 아니라 각 단계에 — 그리고 CI/CD
시크릿(Secret Manager, 키리스 인증), 최소 권한 SA, CI 게이트(lint/test/시크릿/SAST/이미지), CD(WIF), 알림("받으면 행동할 것에만").
현업 추가사항: 감사 로그, 데이터 접근 승인 절차, 장애 대응 런북과 온콜.
→ 예시: [.github/workflows/](../.github/workflows/), [docs/iam.md](iam.md), [terraform/monitoring.tf](../terraform/monitoring.tf)

## 관통하는 원칙 5 (폴더가 그 증거)

1. **멱등성** — 재실행이 안전해야 장애가 무섭지 않다 (ingestion/, dbt/)
2. **원형 보존 → 단계적 정제** — 문제의 층위를 즉시 안다 (레이크/raw/staging/marts)
3. **최소 권한** — 규칙을 문서가 아닌 IAM으로 강제 (docs/iam.md)
4. **전부 코드** — 인프라·변환·스케줄·품질·배포가 리뷰/재현 가능 (terraform/, dbt/, airflow/, .github/)
5. **비용 의식** — 상시 리소스 0 기본, 비싼 것은 실측으로 판단 (PLAN 6장, docs/labs/)

## 현업과 이 프로젝트의 결정적 차이 (면접에서 먼저 말하면 좋은 것)

- **사람**: 현업의 절반은 협상이다 — 원천 소유팀, 소비자 팀, 보안팀과의 합의. 여기선 전부 내가 결정했다
- **레거시**: 그린필드는 드물다. 보통은 "돌아가고 있는 엑셀/크론잡"을 무중단으로 대체하는 일
- **규모**: 수만 건과 수십억 건은 다른 세계 — 다만 원칙(멱등·계보·테스트)은 규모와 무관하게 같다
