# 컨슈머 코드 0줄로 만든 스트리밍 파이프라인, 그리고 SQL로 복원한 추천 신호 — Phase 2

## 배경

[3편](003 링크)까지로 배치 파이프라인(수집→적재→변환→Airflow)이 완성됐다. Phase 2의 목표는 두 가지: **실시간 이벤트 스트리밍**과 프로젝트의 최종 산출물인 **추천 API**다. 완성된 지금, 추천 API는 실제 인터넷에 떠 있다:

```bash
curl "https://recommendation-api-bfvsee2ydq-du.a.run.app/recommendations/dry?limit=3"
```

## 데이터가 없다는 문제 — 크롤링 대신 선택한 것

사용자 행동 데이터(누가 뭘 봤고 샀는지)는 크롤링으로 구할 수 있지만 약관 위반 소지가 있다. 대신 **가상 이벤트 생성기**를 만들었다. 꼼수처럼 들리지만 구조는 실제 기업의 클릭스트림과 동일하고, 결정적인 장점이 하나 있다 — **데이터에 "정답"을 심을 수 있다**. 이 얘기는 뒤에서.

카탈로그도 없어서 만들었는데, 재료는 진짜다: 식약처에서 수집한 실제 성분 21,833개 중에서 뽑아 제품 300개를 합성했다(시드 고정 = 재현 가능). 이 덕에 나중에 실제 규제 데이터와 조인이 된다.

## 스트리밍: Dataflow를 안 쓴 이유

JD마다 나오는 Dataflow(Beam)를 쓰고 싶은 유혹이 있었다. 하지만 초당 몇 건짜리 이벤트에 상시 워커(월 수십만 원)는 명백한 과잉이다. 대신 **Pub/Sub의 BigQuery 구독**을 썼다:

```
이벤트 생성기 ──▶ Pub/Sub 토픽 ──▶ BigQuery 구독(관리형) ──▶ raw.events
                                    ↑ 컨슈머 코드 0줄. 구글이 배달·재시도·적재를 다 함
```

검증 결과: 300건 발행 → 300건 도착, 유실 0, 지연 수 초. "기술을 아는 것"만큼 "안 써야 할 때를 아는 것"도 실력이라고 믿는다 — 이 결정은 ADR(Architecture Decision Record)로 남겼다.

### 스트리밍에서 배운 개념 하나: at-least-once

Pub/Sub는 메시지를 "최소 1번" 배달한다. 즉 **가끔 두 번 오는 게 정상**이다. 그래서 모든 이벤트에 `event_id`(uuid)를 박고, dbt staging에서 중복을 제거한다:

```sql
qualify row_number() over (partition by event_id order by publish_time) = 1
```

"exactly-once는 인프라가 주는 게 아니라 소비자가 만드는 것" — 스트리밍 면접의 단골 주제다.

## 추천: ML 없이, 그러나 검증 가능하게

추천 로직은 dbt(SQL)로 만들었다. 포트폴리오의 평가 대상은 모델 정확도가 아니라 파이프라인이니까.

```
score = 조회×1 + 찜×3 + 구매×5 + 평균평점×2
+ 금지 성분 포함 제품은 전원 제외
+ 사용제한 성분은 민감성 피부에게만 제외   ← 실제 식약처 규제 데이터가 일하는 부분
```

### 심어둔 정답을 찾아내는가 — 합성 데이터의 역설적 장점

이벤트 생성기를 만들 때 가상 사용자들이 **자기 피부타입 제품을 3배 더 보고, 맞는 제품에 후한 평점**을 주도록 설계했다. 즉 데이터에 정답을 심었다. 그럼 추천 mart가 그 신호를 복원하는지 정량 검증이 가능하다:

| | 피부타입 매칭률 |
|---|---|
| 무작위 선택 기준선 | ~36% |
| **추천 상위 5** | **72%** |

"합성 데이터로 만든 추천이 의미가 있나요?"라는 질문에 대한 내 답: **정답을 알고 심었기 때문에, 오히려 파이프라인의 정확성을 숫자로 증명할 수 있었다.**

## 서빙: API는 계산하지 않는다

FastAPI + Cloud Run으로 배포했다. 설계 원칙 하나만 꼽자면 **사전 계산(precompute) 패턴** — 추천 계산은 dbt가 새벽에 끝내두고, API는 그 테이블을 읽기만 한다. 응답이 빠르고, 요청당 비용이 거의 없고, API의 권한도 "marts 조회"로 좁힐 수 있다(raw/staging은 IAM 차원에서 접근 불가).

공개 배포의 안전장치: 읽기 전용 SA + Enum·파라미터 바인딩(인젝션 차단) + 최대 인스턴스 1 + 스케일 투 제로(안 쓰면 0원) + 예산 알림.

## 배운 것

- 관리형 서비스를 잘 고르면 코드가 사라진다 (BigQuery 구독 = 컨슈머 0줄)
- 합성 데이터는 "정답을 아는 테스트베드"가 될 수 있다
- 서빙과 계산을 분리하면 성능·비용·보안이 한꺼번에 좋아진다
- 다음 글: Phase 3 — CI에 보안 스캔(gitleaks/Bandit/Trivy) 심기, 모니터링과 알림
