# 주니어 DE 포트폴리오 시작 — 파이프라인보다 예산 알림을 먼저 만든 이유

## 배경

데이터 엔지니어 취업을 준비하면서 포트폴리오 프로젝트를 시작했다. 주제는 **"피부타입별 화장품 추천 데이터 플랫폼"** — 식약처 공공 데이터(배치)와 사용자 행동 이벤트(스트리밍)를 GCP 위에서 처리해 추천 API까지 서빙하는 엔드투엔드 파이프라인이다.

주제보다 먼저 정한 것은 **원칙**이었다. 채용공고(JD)들을 보면 기술 스택 나열 뒤에 항상 비슷한 문장이 붙는다: "비용을 의식하는 분", "운영 경험", "왜 그 기술을 선택했는지 설명할 수 있는 분". 그래서 코드를 쓰기 전에 기획안(PLAN.md)과 작업 규칙(RULES.md)부터 작성했고, 모든 결정을 문서로 남기기로 했다.

## 첫 번째 코드가 예산 알림인 이유

GCP 크레딧은 ₩432,898, 기한은 8월 말. 학생/취준생 커뮤니티에서 가장 흔한 사고가 "실습하다 클러스터 안 끄고 잠들어서 크레딧 증발"이다. 그래서 **리소스를 하나라도 만들기 전에 예산 알림부터** 걸었다.

여기서 삽질 하나. `gcloud billing budgets create`로 예산을 만들면 기본이 **"크레딧 차감 후 청구액" 기준**이다. 크레딧이 남아있는 동안 청구액은 계속 0원이므로, **크레딧이 다 타버릴 때까지 알림이 한 번도 안 온다.** 감시 장치가 정작 감시를 안 하는 셈이다.

```bash
# 크레딧 '소진 속도' 자체를 감시하려면 gross 기준으로 바꿔야 한다
gcloud billing budgets update <BUDGET_ID> \
  --credit-types-treatment=exclude-all-credits
```

배분 전략도 세웠다: 잔액을 두 달로 나눈 월 ₩216,000을 "이 선에 닿으면 뭔가 잘못됐다"는 알림선으로 쓰고, 실제 목표 지출은 월 2~5만 원으로 잡았다. 아낀 크레딧은 GKE·Dataproc처럼 비싼 기술을 "만들고 → 문서화하고 → 즉시 삭제"하는 실습 버퍼로 쓴다.

## 인프라는 전부 Terraform으로

콘솔 클릭으로 만든 리소스는 ① 재현 불가 ② 리뷰 불가 ③ 비용 추적 불가라서, 부트스트랩(프로젝트 생성, state 버킷) 딱 두 가지만 수동으로 하고 나머지는 전부 코드로 만들었다. 첫 apply 결과는 리소스 6개: 데이터 레이크 버킷, BigQuery raw 데이터셋, 그리고 최소 권한만 가진 수집용 서비스 계정.

서비스 계정 권한은 의도적으로 좁혔다. 프로젝트 editor 하나 주면 편하지만, 수집기가 뚫리면 프로젝트 전체가 뚫린다:

- `storage.objectAdmin` — **데이터 레이크 버킷 하나에만**
- `bigquery.dataEditor` — **raw 데이터셋에만** (가공 영역은 못 건드림)
- `bigquery.jobUser` — 적재 작업 실행용 (이 역할만 프로젝트 단위 부여가 강제라 예외)

## 이번 주 삽질 모음

1. **`.gitignore` 인라인 주석**: `*.tfvars  # 주석` 처럼 패턴 뒤에 주석을 붙이면 주석까지 패턴의 일부가 되어 무시가 안 된다. 시크릿이 들어가는 tfvars가 커밋될 뻔했다. `git check-ignore -v <파일>`로 검증하는 습관을 얻었다.
2. **`.env.*` 패턴이 `.env.example`까지 삼킴**: 커밋돼야 할 예시 파일이 무시됐다. `!.env.example` 예외 규칙으로 해결.
3. **공공데이터포털 오퍼레이션 경로**: 승인받은 엔드포인트 URL로 바로 호출하면 500이 난다. 실제로는 `{서비스}/{오퍼레이션}` 형태라서, 후보 패턴을 코드로 순회하며 찾아냈다. 세 API의 오퍼레이션 명명이 제각각(`get...Service01`, `get...Service`, `get...InfoList01`)이라 "공공 API는 문서보다 실호출로 검증"이라는 교훈을 얻었다.

## 배운 것
