# 포트폴리오 소개 — Beauty Pipeline (1페이지)

> **사용법**: 이력서 첨부, 지원서 "포트폴리오 설명" 칸, 면접 사전자료로 복붙하는 문서.
> Phase가 끝날 때마다 [진행 상황]과 [성과] 숫자를 갱신한다. `(예정)` 표시는 완료 시 제거.

## 피부타입별 화장품 추천 데이터 플랫폼

**GitHub**: https://github.com/donghajang213/beauty-pipeline · **기간**: 2026.07 ~ (진행 중) · **인원**: 1인

### 한 줄 요약

식약처 공공 데이터(배치)와 사용자 행동 이벤트(스트리밍)를 GCP에서 처리해 피부타입별 화장품 추천 API로 서빙하는 엔드투엔드 데이터 플랫폼. 인프라 전체를 Terraform으로 관리하고 CI에 보안 스캔을 내장.

### 기술 스택

`Python` `SQL` `BigQuery` `GCS` `Pub/Sub` `Airflow` `dbt` `FastAPI` `Terraform` `Docker` `Cloud Run` `GitHub Actions` `gitleaks/Bandit/Trivy`

### 아키텍처

```
식약처 API ─[배치/Airflow]→ GCS(raw) → BigQuery → dbt(성분·피부타입 mart) → 추천 API(FastAPI/Cloud Run)
가상 사용자 이벤트 ─[스트리밍]→ Pub/Sub → BigQuery ↗            (Terraform IaC / CI·CD / 예산·장애 알림)
```

### 핵심 성과 (진행하며 갱신)

- 인프라 100% 코드화: 부트스트랩 2건 외 모든 리소스를 Terraform으로 생성, `apply` 1회로 재현
- 최소 권한 IAM 설계: 서비스 계정별 권한을 버킷/데이터셋 단위로 제한, 근거를 권한 대장으로 문서화
- 비용 설계: 크레딧 소진 속도 기준(gross) 예산 알림 이원화 — 운영 지출 월 2~5만 원 이내 (예정: 실측치)
- 식약처 API 3종(총 6.1만 건) 수집 검증 — 재시도·페이지네이션 공용 클라이언트, 단위 테스트 커버
- 배치 적재 멱등성 설계: 스냅샷+날짜 파티션 교체(WRITE_TRUNCATE) — 실제 3회 네트워크 단절을 "재실행"만으로 복구
- dbt staging 계층: 프로파일링 근거 기반 품질 테스트 (중복 1건 탐지·제거, grain 분석으로 false alarm 방지)
- CI 자동화: PR마다 ruff lint/format + pytest를 병합 게이트로 (GitHub Actions)
- Airflow 일 배치 DAG: 검증 중 실제 장애 5건(PC 종료, 회선 단절 2회 등)을 재시도·재개로 자동 극복 (사람 개입 1회)
- 스트리밍: Pub/Sub BigQuery 구독(컨슈머 코드 0줄) — 발행 300건 유실 0, event_id 중복 제거로 at-least-once 대응
- 추천 mart 정량 검증: 합성 데이터에 심은 선호 신호를 SQL 집계로 복원 — 상위5 매칭률 **72% vs 무작위 36%**
- **추천 API 라이브 서빙**: FastAPI + Cloud Run 스케일 투 제로, marts 전용 최소 권한 SA
  → https://recommendation-api-bfvsee2ydq-du.a.run.app/docs
- CI 보안 게이트 3종: gitleaks(이력 유출 0) · Bandit(오탐 판단 근거 문서화) · Trivy(HIGH/CRITICAL 0) — 병합 차단 조건
- 시크릿 무결성: Secret Manager 이전 (값은 TF state에 남기지 않는 절차 설계) + 환경변수 폴백 체인
- 키리스 CD: Workload Identity Federation — 저장소 한정 OIDC 신뢰, SHA 태그 배포, 스모크 테스트 포함
- 운영 알림: uptime check(5분)·5xx 즉시·예산 4단계 — "받으면 행동할 것에만" 원칙으로 소음 차단

### 의사결정 하이라이트 (면접 어필 포인트)

| 결정 | 이유 |
|---|---|
| 리뷰 크롤링 배제 → 합성 이벤트 생성기 | 약관 위반 리스크 제거. 실제 클릭스트림과 동일한 스키마 설계 경험 확보 |
| Kafka 대신 Pub/Sub | 동일 개념(pub/sub 모델)을 관리형으로 — 1인·한정 크레딧에서 운영 부담 최소화 |
| Cloud Composer 대신 로컬 Docker Airflow | Composer 상시 비용(월 수십만 원)이 프로젝트 예산과 불균형 |
| 추천을 ML이 아닌 dbt/SQL로 | 평가 대상은 파이프라인. 모델은 mart 교체만으로 확장 가능한 구조로 유보 |
