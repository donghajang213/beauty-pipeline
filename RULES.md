# 📏 RULES — 프로젝트 작업 규칙

> **이 문서의 역할**: 프로젝트를 진행하는 동안 사람(나)과 AI(Claude)가 함께 지켜야 할 규칙.
> 규칙이 필요한 이유: 주니어 포트폴리오의 신뢰도는 "결과물"보다 **"일하는 방식"** 에서 드러난다.
> 규칙을 어겨야 할 상황이 오면, 어기기 전에 이 문서를 먼저 수정하고 이유를 기록한다.

---

## 1. 문서화 규칙 (가장 중요)

### 1.1 모든 디렉터리에 README.md
새 디렉터리를 만들면 반드시 README.md를 함께 만들고 다음을 적는다.
- **무엇**: 이 폴더에 뭐가 들어있는가
- **왜**: 왜 이 구조/기술을 선택했는가
- **어떻게**: 실행/테스트 방법

### 1.2 모든 파일 상단에 설명 주석
새 코드 파일을 만들 때 파일 최상단에 한국어 주석으로 적는다.
```python
"""
파일: ingestion/batch/fetch_cosmetics.py
역할: 식약처 화장품 API에서 제품/성분 데이터를 받아 GCS raw 버킷에 저장
입력: 공공데이터포털 식약처 화장품 API (인증키는 Secret Manager)
출력: gs://{bucket}/raw/cosmetics/dt={날짜}/data.json
실행: python -m ingestion.batch.fetch_cosmetics --date 2026-07-06
"""
```

### 1.3 결정 기록 (ADR)
기술 선택·아키텍처 변경 등 중요한 결정은 `docs/adr/` 에 기록한다.
- 형식: `NNN-제목.md` (예: `001-pubsub-vs-kafka.md`)
- 내용: 상황 → 선택지 → 결정 → 이유 → 트레이드오프
- **면접에서 "왜 그 기술을 썼나요?"에 대한 답변이 여기서 나온다.**

### 1.4 트러블슈팅 기록
30분 이상 헤맨 문제는 해결 후 `docs/troubleshooting/`에 기록한다.
- 증상 → 원인 → 해결 → 배운 점

## 2. Git 규칙

- **커밋 단위**: 하나의 커밋은 하나의 논리적 변경만. "여러 작업 몰아서 커밋" 금지.
- **커밋 메시지**: Conventional Commits 형식
  - `feat: 화장품 제품 데이터 배치 수집기 추가`
  - `fix: BigQuery 적재 시 timezone 오류 수정`
  - `docs:`, `refactor:`, `test:`, `chore:`, `ci:` 등
- **브랜치**: `main`은 항상 동작하는 상태 유지. 기능 작업은 `feat/이름` 브랜치 → PR → 병합.
  (혼자 하는 프로젝트라도 PR 흐름을 유지한다 — CI가 도는 모습 자체가 포트폴리오다.)
- **금지**: `main`에 force push, 시크릿 커밋(아래 4.1 참조)

---

## 3. 코드 규칙

- **언어**: Python 3.12 고정. 패키지 관리는 `uv` 또는 `pip + requirements.txt` 하나로 통일.
- **스타일**: `ruff` (lint + format) 통과 필수. CI에서 강제.
- **타입 힌트**: 함수 시그니처에는 반드시 타입 힌트 작성.
- **테스트**: 비즈니스 로직(변환, 파싱)은 단위 테스트 작성. 외부 API 호출은 mock 처리.
- **하드코딩 금지**: 버킷명, 프로젝트 ID, 날짜 등은 환경변수/설정 파일/인자로 주입.
- **멱등성(idempotency)**: 모든 배치 작업은 같은 날짜로 재실행해도 결과가 중복되지 않아야 한다.
  (데이터 엔지니어링 면접 단골 주제 — 재실행 = 덮어쓰기 설계)

## 4. 보안 규칙 (DevSecOps)

### 4.1 시크릿
- API 키, 서비스 계정 키, 비밀번호는 **절대 Git에 커밋하지 않는다.**
- 로컬: `.env` 파일 (반드시 `.gitignore`에 포함) / 클라우드: **Secret Manager**
- 커밋 전 `gitleaks`가 CI에서 검사한다. 실수로 커밋했다면 즉시 키를 **폐기·재발급**한다 (히스토리 삭제만으로는 부족).

### 4.2 IAM
- 서비스 계정은 **역할(용도)별로 분리**하고 최소 권한만 부여한다.
  - 예: `sa-ingestion`(GCS 쓰기만), `sa-dbt`(BigQuery 데이터 편집만)
- `roles/owner`, `roles/editor` 같은 광범위 역할을 서비스 계정에 부여하지 않는다.
- 부여한 권한과 이유를 `docs/iam.md`에 기록한다.

### 4.3 CI 보안 게이트
- PR마다: `gitleaks`(시크릿 스캔) + `bandit`(Python SAST) + `trivy`(Docker 이미지 스캔)
- 스캔 실패 시 병합 불가.

## 5. 비용 규칙 (크레딧 보호)

1. **예산 알림이 없으면 리소스를 만들지 않는다.** (Phase 0 최우선 작업)
2. 상시 실행 리소스(VM, 클러스터)는 만들기 전에 **월 예상 비용을 먼저 계산**하고 PLAN.md 비용 표와 비교한다.
3. 실험용 리소스는 그날 안에 삭제한다. Terraform 밖에서 콘솔로 "손으로" 만든 리소스는 금지 (추적 불가 = 비용 누수).
4. BigQuery 쿼리는 실행 전 예상 스캔량 확인. 원본 테이블은 날짜 파티션 필수.
5. 주 1회 결제 대시보드를 확인하고 `docs/cost-log.md`에 기록한다.

---

## 6. 인프라 규칙 (IaC)

- 모든 GCP 리소스는 **Terraform으로만** 생성한다. 콘솔 수동 생성 금지.
  - 예외: GCP 프로젝트 생성, 결제 연결 등 최초 부트스트랩만 수동 허용 (docs에 기록)
- `terraform plan` 결과를 확인하지 않고 `apply` 하지 않는다.
- Terraform state는 GCS 백엔드에 저장한다 (로컬 state 파일 커밋 금지).

## 7. AI(Claude) 협업 규칙

이 프로젝트의 목적은 **취업이 아니라 취업할 실력을 만드는 것**이다. AI는 가속 도구이지 대체물이 아니다.

1. **설명 우선**: Claude는 코드를 작성할 때 "무엇을, 왜 그렇게" 하는지 반드시 설명한다. 특히 처음 등장하는 개념(파티셔닝, 멱등성, DAG 등)은 짧게라도 개념 설명을 덧붙인다.
2. **한 번에 하나씩**: 한 세션에서 여러 Phase를 몰아치지 않는다. 사용자가 각 단계를 이해하고 직접 실행해본 뒤 다음으로 넘어간다.
3. **면접 관점 체크**: 주요 구현이 끝나면 "이 부분에서 나올 수 있는 면접 질문"을 함께 정리한다.
4. **비용 발생 작업은 사전 고지**: GCP 리소스를 만드는 작업 전에는 예상 비용을 먼저 알려준다.
5. **PLAN.md / RULES.md 준수**: 기획과 다른 방향이 필요하면 코드보다 문서를 먼저 수정한다.

## 8. 주간 루틴

- **매주 일요일**: 진행 상황을 PLAN.md 로드맵 체크박스에 반영 + `docs/weekly/` 에 짧은 회고 (한 일 / 배운 것 / 다음 주 계획 / 비용 확인)
- 회고는 3~5줄이면 충분하다. 꾸준함이 완성도보다 중요하다.
